# 23. HTTP: как браузер разговаривает с сервером

Когда вы открываете сайт, браузер отправляет запрос на сервер и получает ответ. Этот обмен происходит по правилам протокола HTTP. Понимать эти правила важно — они лежат в основе любого API, бота и веб-сервиса.

## Запрос и ответ

HTTP работает по схеме «запрос — ответ»:

1. **Клиент** (браузер, `curl`, приложение) отправляет запрос серверу
2. **Сервер** обрабатывает запрос и возвращает ответ

Это всегда инициирует клиент — сервер сам ничего не присылает.

## Структура URL

Каждый запрос адресован конкретному URL:

```
https://api.example.com/users?role=admin
│       │               │     │
схема   хост            путь  параметры
```

- **схема** — `http` или `https` (с шифрованием)
- **хост** — адрес сервера
- **путь** — что именно запрашиваем на сервере
- **параметры** — дополнительные данные после `?`, разделённые `&`

Параметры передаются серверу как аргументы: `?city=Caerleon&quality=1`.

## Статус-коды

Каждый ответ содержит числовой код — сервер сообщает, что произошло:

**2xx — успех**, запрос выполнен:

| Код | Смысл |
|-----|-------|
| **200** | OK — всё хорошо, вот данные |
| **201** | Created — ресурс создан (обычно в ответ на POST) |

**3xx — редирект**, нужно идти дальше:

| Код | Смысл |
|-----|-------|
| **301** | Moved Permanently — адрес изменился навсегда |
| **302** | Found — временный редирект |

**4xx — ошибка клиента**, запрос составлен неправильно или нет доступа:

| Код | Смысл |
|-----|-------|
| **400** | Bad Request — запрос составлен неправильно |
| **401** | Unauthorized — нужна авторизация |
| **403** | Forbidden — авторизован, но доступ запрещён |
| **404** | Not Found — такого адреса нет |

**5xx — ошибка сервера**, клиент всё сделал правильно:

| Код | Смысл |
|-----|-------|
| **500** | Internal Server Error — ошибка в коде на сервере |
| **502** | Bad Gateway — сервер не смог получить ответ от другого сервиса |

Стандартные коды заканчиваются на 511, но разработчики иногда добавляют свои. Несколько известных:

| Код | Смысл |
|-----|-------|
| **418** | I'm a teapot — «я чайник, не могу варить кофе». Появился в RFC 2324 как первоапрельская шутка в 1998 году. Официально существует до сих пор. |
| **451** | Unavailable For Legal Reasons — доступ заблокирован по требованию властей. Назван в честь романа Рэя Брэдбери «451 градус по Фаренгейту». |
| **410** | Gone — как 404, но сервер специально сообщает: «было, но я удалил навсегда». |
| **429** | Too Many Requests — слишком много запросов за короткое время. Именно этот код вернёт Albion API, если запрашивать цены слишком часто. |

## Заголовки

Вместе с запросом и ответом передаются заголовки — метаданные об этом сообщении. Несколько важных:

**В запросе:**
- `Content-Type: application/json` — я отправляю JSON
- `Authorization: Bearer токен` — мой токен доступа
- `User-Agent: curl/7.68` — кто делает запрос

**В ответе:**
- `Content-Type: application/json` — сервер вернул JSON
- `Content-Length: 1234` — размер ответа в байтах

## Смотрим реальные запросы в браузере

Откройте любой браузер и нажмите **F12** (или Ctrl+Shift+I). Перейдите на вкладку **Network**.

Теперь откройте в новой вкладке:

```
https://httpbin.org/get
```

Httpbin — это сервис для изучения HTTP: он принимает запросы и возвращает в ответе всё, что получил.

В DevTools во вкладке Network появится запрос. Кликните на него. Вы увидите:

- **Headers** — заголовки запроса и ответа
- **Response** — тело ответа (JSON)
- **Preview** — то же самое, но читаемее

В теле ответа будет объект с вашими заголовками, IP-адресом и URL. Это именно то, что увидел сервер.

## Смотрим разные статус-коды

Откройте в браузере — и каждый раз смотрите в Network на код ответа:

**200 — успех:**
```
https://httpbin.org/status/200
```

**404 — не найдено:**
```
https://httpbin.org/status/404
```

**500 — ошибка сервера:**
```
https://httpbin.org/status/500
```

**301 — редирект:**
```
https://httpbin.org/redirect/1
```

При редиректе в Network появятся два запроса: первый со статусом 301, второй — итоговый.

## Шпаргалка

- HTTP — протокол запрос/ответ между клиентом и сервером.
- URL: схема + хост + путь + параметры.
- 2xx — успех, 4xx — ошибка клиента, 5xx — ошибка сервера.
- F12 → Network — смотреть реальные запросы в браузере.
- httpbin.org — сервис для изучения и тестирования HTTP.

## Напоследок

- HTTP расшифровывается как HyperText Transfer Protocol. Первая версия HTTP/1.0 появилась в 1996 году — тогда он передавал только HTML-страницы. Сегодня по HTTP передаётся всё: JSON, изображения, видео.
- HTTPS — это HTTP с шифрованием через TLS. Сервер и клиент договариваются о ключе шифрования до того, как начать обмен данными, поэтому перехватить содержимое запроса невозможно.

## Практика

23.1. Откройте DevTools (F12 → Network) и перейдите на `https://httpbin.org/get`. Найдите в ответе поле `"url"`.

   В теле ответа должно быть `"url": "https://httpbin.org/get"`.

23.2. Откройте `https://httpbin.org/status/404`. Посмотрите статус в колонке Status в Network.

   Должна появиться строка с красным кодом 404.

23.3. Откройте `https://httpbin.org/redirect/1` и посмотрите, сколько запросов появилось в Network.

   Должны появиться два запроса: первый со статусом 302, второй — 200.
